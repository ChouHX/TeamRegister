下面是按当前实际验证结果整理的 **支付链接获取 / 支付确认链路**。

重点修正：

- `checkout_session_id` 的第一来源不是 `elements/sessions`
- 第一跳应该是 `POST /backend-api/payments/checkout`
- `elements/sessions`、`payment_pages/init`、`payment_pages/{cs}` 都属于 **前端组件初始化链路**
- 我们当前走的是 **协议直连链路**
- 协议直连时，不需要把前端组件初始化接口当主链路前置步骤

---

# 一、协议主链路

```
1  创建 Checkout Session
POST https://chatgpt.com/backend-api/payments/checkout
↓
2  确认支付
POST https://api.stripe.com/v1/payment_pages/{checkout_session_id}/confirm
```

---

# 二、核心结论

## 1. `checkout_session_id` 的正确来源

正确来源：

```text
POST /backend-api/payments/checkout
```

不是：

```text
GET /v1/elements/sessions
```

---

## 2. `checkout_session_id` 在单次支付流程里保持不变

也就是：

```text
第一次拿到后，本轮支付流程内一直复用
```

所以：

- 不需要重复获取
- 不需要每次再走一遍 `elements/sessions`

---

## 3. `elements/sessions` 是什么

它返回的是一个 **Stripe 前端支付组件 session**。

用途：

- 给前端页面里的 Stripe Elements 组件初始化用
- 帮浏览器页面渲染支付组件

它不是我们协议流里必须的第一步。

---

## 4. `payment_pages/init` 和 `payment_pages/{cs}` 是什么

这两个接口本质上也是：

```text
前端组件初始化
```

也就是说它们主要服务：

- 页面加载
- Stripe 支付组件挂载
- 前端状态初始化

我们现在走协议，不走前端组件，所以这两个接口不是主链路必需步骤。

---

# 三、接口1：创建 Checkout Session

### 接口

```http
POST https://chatgpt.com/backend-api/payments/checkout
```

### 作用

```text
创建本次支付流程对应的 checkout_session_id
```

### 关键请求头

| 参数 | 来源 |
| --- | --- |
| authorization | 登录流程返回的 `access_token` |
| __Secure-next-auth.session-token | 登录态 cookie |
| __Host-next-auth.csrf-token | 登录态 cookie |
| oai-device-id | 登录流程里的 `oai-did` |
| openai-sentinel-token | 支付场景动态生成 |
| cf 系列 cookie | 当前登录会话里已有 |
| oai-client-version | ChatGPT Web 客户端版本 |
| oai-client-build-number | ChatGPT Web 构建号 |

### 关键请求体

```json
{
  "plan_name": "chatgptteamplan",
  "team_plan_data": {
    "workspace_name": "Artizancloud",
    "price_interval": "month",
    "seat_quantity": 5
  },
  "billing_details": {
    "country": "JP",
    "currency": "JPY"
  },
  "cancel_url": "https://chatgpt.com/?promo_campaign=team1dollar#team-pricing",
  "promo_campaign": {
    "promo_campaign_id": "team1dollar",
    "is_coupon_from_query_param": true
  },
  "checkout_ui_mode": "custom"
}
```

### 关键返回

常见返回字段：

```text
checkout_session_id
session_id
checkout_url
client_secret
```

其中我们最关心的是：

```text
checkout_session_id
```

---

# 四、接口2：确认支付

### 接口

```http
POST https://api.stripe.com/v1/payment_pages/{checkout_session_id}/confirm
```

### 作用

```text
直接提交支付确认
```

### 当前协议链结论

我们当前协议流可以直接走：

```text
checkout -> confirm
```

不再要求先走：

```text
elements/sessions
payment_pages/init
payment_pages/{checkout_session_id}
```

---

# 五、前端组件链路说明

下面这些接口依然存在，但属于 **前端组件初始化链路**：

```http
GET  https://api.stripe.com/v1/elements/sessions
POST https://api.stripe.com/v1/payment_pages/{checkout_session_id}/init
POST https://api.stripe.com/v1/payment_pages/{checkout_session_id}
```

它们的作用是：

- 渲染前端支付组件
- 初始化前端 Stripe 页面状态
- 返回组件级别的 session 参数

对于我们当前的协议实现：

```text
可以请求
但不是必须请求
```

尤其是：

```text
checkout_session_id 不是从这里拿
```

# 七、我们当前实现应该怎么走

正确顺序：

```text
注册 / 登录
↓
拿到 access_token + 登录态 cookie + oai-did
↓
生成支付专用 openai-sentinel-token
↓
POST /backend-api/payments/checkout
↓
拿到 checkout_session_id
↓
GET https://m.stripe.com/6
↓
拿到 guid / muid / sid
↓
POST /v1/payment_pages/{checkout_session_id}/confirm
```

# 八、协议实现里的注意点

## 1. 不要再把 `elements/sessions` 当第一步

旧认知：

```text
elements/sessions -> checkout_session_id
```

现在修正为：

```text
/backend-api/payments/checkout -> checkout_session_id
```

---

## 2. 不要把 `init` / `payment_pages` 当主链路前置

旧认知：

```text
checkout_session_id -> init -> payment_pages -> confirm
```

现在修正为：

```text
checkout_session_id -> confirm
```

---

## 3. 前端接口不是没用，而是对我们当前协议流不是刚需

也就是：

- 前端页面仍然可能会调用它们
- 但我们协议直连时不需要依赖它们完成主链路

# 十、结论

一句话总结：

```text
checkout_session_id 的真正第一来源是 ChatGPT 的 /backend-api/payments/checkout，
不是 Stripe 的 elements/sessions。
```

所以我们现在的协议主链路应该是：

```text
checkout -> confirm
```

而不是：

```text
elements -> init -> payment_pages -> confirm
```
